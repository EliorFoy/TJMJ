\small\textbf{Abstract}\quad
This paper presents TJMJ, a comprehensive full-stack web-based implementation of the Taojiang classic Wang Mahjong (桃江经典王麻将), a regional variant of Chinese mahjong originating from Hunan Province. The platform integrates a Vue~3 Composition API single-page frontend with a Node.js WebSocket game server, supporting three operational modes: single-player AI, four-player real-time multiplayer, and spectator observation. Key technical contributions include: (1) a complete mathematical formalization of the Taojiang rule set encompassing dice-driven wall breaking, dynamic joker (\textit{wang}) determination, and an additive scoring model with 14 distinct winning patterns; (2) a recursive backtracking hu-detection engine with $\mathcal{O}(3^k)$ worst-case complexity and sub-millisecond average runtime; (3) a production-grade multiplayer infrastructure featuring exponential-backoff reconnection, server-authoritative state synchronization, and 10-second action timeout deadlock prevention; (4) a Agora-based real-time voice subsystem with Agora SD-RTN relay (10\,GB/month free tier) and incremental peer-connection management; (5) cross-platform compatibility solutions addressing iOS Safari click events, AudioContext suspension, Android dynamic viewport units, and CSS-transform-induced \texttt{position:fixed} failures. The complete system comprises approximately 4200 lines of JavaScript across 12 core files, with the frontend deployed via GitHub Pages and the backend exposed through ngrok TCP tunneling. This report provides exhaustive documentation of the system architecture, communication protocols, algorithmic implementations, and device-compatibility engineering.

\section{Introduction}
% ===================================================================

Chinese mahjong, originating in the Qing dynasty, has evolved into numerous regional variants distinguished by tile composition, scoring mechanisms, and special rules. Among these, the Taojiang (桃江) variant---''Classic Wang Mahjong''---is characterized by a simplified 108-tile set (excluding wind and flower tiles), a dynamic joker mechanism (\textit{dasi banwang}, 打筛扳王), a post-victory multiplier system (\textit{zhaniao}, 扎鸟), and an additive scoring model where multiple winning patterns (\textit{menzi}, 门子) accumulate multiplicatively.

Despite the global popularity of mahjong, existing digital platforms fail to adequately serve the Taojiang community. Commercial applications such as Xianyi (闲逸) require mandatory registration, feature cluttered interfaces, deviate from authentic local rules, and lack modern interaction modalities including drag-to-reorder hand tiles, real-time voice communication, and text-based danmaku chat. Furthermore, no existing platform provides an open-source foundation for training mahjong artificial intelligence agents.

TJMJ was conceived to address these gaps as a fully open-source, zero-installation, browser-based mahjong platform. The system faithfully implements the complete Taojiang rule set while leveraging modern web technologies---Vue~3 Reactive Composition API, Vite~8 build toolchain, Node.js WebSocket servers, and Agora peer-to-peer communication---to deliver a responsive, cross-platform gaming experience.

\textbf{Contributions.} The primary contributions of this work are:
\begin{enumerate}[nosep]
  \item A rigorous mathematical formalization of the Taojiang mahjong rule set, including tile encoding, dice mechanics, joker determination, action priority, and multiplicative scoring;
  \item A production-grade full-stack architecture achieving sub-100ms state synchronization latency through server-authoritative game logic and JSON-based WebSocket messaging;
  \item A recursive backtracking hu-detection algorithm with empirical $<1$\,ms runtime on commodity hardware;
  \item A fault-tolerant multiplayer infrastructure featuring exponential-backoff WebSocket reconnection, server-side action timeout deadlock prevention, and URL-parameter-based room state persistence;
  \item Comprehensive cross-platform compatibility engineering addressing iOS, Android, and desktop-specific rendering and interaction challenges.
\end{enumerate}

\textbf{Paper Organization.} Section~2 surveys related work. Section~3 presents the system architecture. Section~4 formalizes the game rules. Section~5 details core algorithms. Section~6 describes the multiplayer and voice subsystems. Section~7 discusses cross-platform compatibility. Section~8 reports performance measurements. Section~9 concludes with future directions.

\subsection{Regional Mahjong Variants}

Chinese mahjong exhibits substantial regional diversity. Standard competition mahjong (国标麻将) employs a 144-tile set with 81 scoring patterns, while Sichuan Bloody Mahjong (四川血战麻将) uses 108 tiles with three-suit composition and mandatory discard-after-kong rules. The Taojiang variant studied herein is closest to the Yiyang (益阳) regional style, distinguished by: (1) the \textit{wang} (王) joker mechanism determined through dice-roll wall-breaking; (2) the \textit{zhaniao} post-victory multiplier system; (3) additive rather than multiplicative scoring; and (4) the ``hard'' versus ``soft'' distinction based on joker presence.

\subsection{Web Technologies for Real-Time Gaming}

Modern browser APIs have enabled sophisticated real-time multiplayer experiences without native installation. The WebSocket protocol (RFC~6455) provides full-duplex communication with minimal overhead, while Agora SD-RTN provides cloud-based real-time voice with automatic NAT traversal. These technologies collectively form the foundation of TJMJ's architecture.

\section{Game Rules \& Mathematical Formulation}
% ===================================================================

\subsection{Tile Encoding Scheme}

Let $\mathcal{T} = \{11,\ldots,19,21,\ldots,29,31,\ldots,39\}$ denote the complete tile set of $|\mathcal{T}|=108$ tiles. Each tile $t \in \mathcal{T}$ is encoded as $t = 10 \cdot s + v$, where $s \in \{1,2,3\}$ denotes the suit (1=Wan/万, 2=Tiao/条, 3=Tong/筒) and $v \in \{1,\ldots,9\}$ denotes the face value. Each distinct tile appears with multiplicity 4.

The subset of \textit{jiang} (将) tiles is defined as:
\begin{equation}
  \mathcal{J} = \{t \in \mathcal{T} \mid t \bmod 10 \in \{2,5,8\}\}
\end{equation}
with $|\mathcal{J}| = 3 \times 3 \times 4 = 36$.

\subsection{Dice Mechanics \& Initial Dealing}

Let $d_1, d_2 \sim \text{Uniform}\{1,\ldots,6\}$ be independent dice rolls, ordered such that $d_1 \leq d_2$. The dealer (庄家) position $D \in \{0,1,2,3\}$ is determined by the previous round's winner. The starting wall belongs to player $P_k$ where:
\begin{equation}
  k = (D + d_1 + d_2 - 1) \bmod 4
\end{equation}

The initial dealing sequence distributes 53 tiles as follows: for iteration $i=0,\ldots,52$, player $p(i)$ receives tile $i$, where:
\begin{equation}
  p(i) = \begin{cases}
    i \bmod 4, & 0 \leq i \leq 47 \quad\text{(3 rounds of 4 players $\times$ 4 tiles)} \\
    i - 48,    & 48 \leq i \leq 51 \quad\text{(dealer supplement)} \\
    0,         & i = 52 \quad\text{(dealer's 14th tile)}
  \end{cases}
\end{equation}
Result: dealer holds 14 tiles, three non-dealers hold 13 tiles each.

\subsection{Wang (Joker) Determination}

The \textit{di} (地) tile $T_d$ is exposed from the wall at stack index determined by the larger die $d_2$, counting from the left side of the next player's wall. The \textit{wang} (王, joker) tile $T_w$ is then computed as:
\begin{equation}
  T_w = \begin{cases}
    \lfloor T_d/10 \rfloor \times 10 + 1, & \text{if } T_d \bmod 10 = 9 \\
    T_d + 1, & \text{otherwise}
  \end{cases}
\end{equation}

This cyclic mapping ensures that if a 9 is exposed, the joker wraps to the 1 of the same suit.

\subsection{Action Priority Model}

When a player $P_s$ discards tile $t$, each other player $P_i$ ($i \neq s$) may potentially intercept. The set of legal actions for player $i$ is:
\begin{equation}
  \mathcal{A}_i(t) = \{a \in \{\text{HU},\text{GANG},\text{PENG},\text{CHI}\} \mid \text{legal}(i,t,a)\}
\end{equation}

The global priority ordering is enforced as:
\begin{equation}
  \text{HU} \succ \text{GANG} \succ \text{PENG} \succ \text{CHI}
\end{equation}

If multiple players can perform the same action, the player closest in counter-clockwise order from the discarder has priority. Players are notified via \texttt{action\_prompt} messages and must respond within a 10-second server-side timeout, after which the action expires.

\subsection{Winning Pattern Classification}

A hand $H$ is a winning hand (\textit{hu pai}, 胡牌) if it can be partitioned into one \textit{jiang} pair $J = \{t_j, t_j\}$ where $t_j \in \mathcal{J}$, and four \textit{sentences} $S_1,\ldots,S_4$, where each $S_k$ is either a triplet (\textit{kezi}, 刻子) $\{t,t,t\}$ or a sequence (\textit{shunzi}, 顺子) $\{t,t+1,t+2\}$ with identical suit. Joker tiles ($T_w$) may substitute for any tile in this partition.

Let $\text{wang}(H)$ denote the number of joker tiles in $H$. Table~\ref{tab:hu_type} classifies winning patterns by their scoring multiplier $m$.

\subsection{Scoring Model}

The total score $\Sigma$ for a self-drawn win (\textit{zimo}, 自摸) is computed as:
\begin{equation}
  \Sigma = B \cdot \left(\sum_{k} m_k\right) \cdot Z \cdot G
\end{equation}
where $B=3$ is the base score, $\sum m_k$ is the sum of all applicable pattern multipliers, $Z\in\{1,2\}$ is the \textit{zhaniao} multiplier, and $G\in\{1,2\}$ is the kong multiplier.

The \textit{zhaniao} (扎鸟) mechanism operates post-victory: a tile $T_z$ is drawn from the wall, and its face value $v = T_z \bmod 10$ determines which losing players have their losses doubled:
\begin{equation}
  Z_i = \begin{cases}
    2, & \text{if } v \in \{1,5,9\} \quad\text{(all players)}\\
    2, & \text{if } v \in \{2,6\} \text{ and } i = (W+1)\bmod 4 \quad\text{(next player)}\\
    2, & \text{if } v \in \{3,7\} \text{ and } i = (W+2)\bmod 4 \quad\text{(opposite)}\\
    2, & \text{if } v \in \{4,8\} \text{ and } i = (W+3)\bmod 4 \quad\text{(previous)}\\
    1, & \text{otherwise}
  \end{cases}
\end{equation}
where $W$ is the winner's position and $i$ indexes each losing player.

\subsection{Hu Detection Algorithm}

Algorithm~\ref{alg:hu} presents the recursive backtracking hu-detection engine.

\begin{figure}[H]
\caption{Hu Detection Algorithm Overview}
\label{alg:hu}
\footnotesize
\begin{verbatim}
Input:  hand H (14 tiles), wang tile Tw, di tile Td, first-turn flag f
Output: {canHu, type, score, hasWang, canCatchCannon}

1.  N = {t in H | t != Tw}                    // normal tiles
    nw = count(t in H where t == Tw)           // joker count
    nd = count(t in N where t == Td)           // di count
    Sort N ascending
2.  IF f AND |H|=14 AND nw=0:                 // black sky hu
      IF no jiang AND no sentence in H: return {T, "黑天胡", 2, F, F}
3.  IF nw=4: return {T, "天天胡", 5, T, F}
    IF nw=3: return {T, "天胡", 3, T, F}
    IF nd=3: return {T, "地胡", 2, T, F}
4.  normalHu = CheckNormalHu(N, nw)
    is7Pairs = Check7Pairs(N, nw)
5.  IF NOT normalHu.canHu AND NOT is7Pairs: return {F, "", 0, nw>0, !(nw>0)}
6.  // Priority-ordered pattern classification:
    isJiangJiang = all t in H: t==Tw OR is258(t)
    isQingYiSe = |{floor(t/10) for t in N}| == 1
    IF isJiangJiang: return {T, "将将胡", 6, ...}
    IF isQingYiSe:   return {T, "清一色", 6, ...}
    IF is7Pairs:      return {T, "七小对", 6, ...}
    IF isPengPeng:    return {T, "碰碰胡", 6, ...}
    IF nw=0:          return {T, "硬庄", 6, ...}
    return {T, "平胡", 3, ...}
\end{verbatim}
\end{figure}

The recursive subroutine \texttt{CheckNormalHu}$(N, w)$ operates by selecting the smallest tile $t_{\min} \in N$ and attempting three reductions: (a) triplet removal of $\{t_{\min}, t_{\min}, t_{\min}\}$ if count $\geq 3$; (b) sequence removal of $\{t_{\min}, t_{\min}+1, t_{\min}+2\}$ if all three exist; (c) jiang pair removal of $\{t_{\min}, t_{\min}\}$ if no jiang has been set yet and count $\geq 2$. When a reduction fails due to missing tiles, $w$ jokers are consumed as substitutes. The recursion depth is bounded by the number of sentences plus jiang ($k \leq 5$), yielding $\mathcal{O}(3^k)$ worst-case complexity and sub-millisecond average runtime on commodity hardware.

\subsection{Ting Detection}

A hand $H$ with $|H| = 3n+1$ is \textit{ting} (听牌) if $\exists t \in \mathcal{T}$ such that $\texttt{CheckHu}(H \cup \{t\})$ succeeds. The \texttt{RuleChecker.isTing()} function enumerates candidates from $\mathcal{T}' \subseteq \mathcal{T}$ where $\mathcal{T}'$ is restricted to tiles within $\pm 2$ face values of tiles already in $H$, reducing the search space from 34 to typically 12--18 candidates. Each candidate check invokes the full hu-detection pipeline. Empirical performance is $<1$\,ms per call.

\subsection{NPC Strategy Agent}

The single-player AI (\texttt{NpcStrategy.js}, 227 lines) employs a heuristic decision framework:

\textbf{Discard Selection.} For each tile $t_i$ in hand $H$, the algorithm computes a composite score:
\begin{equation}
  S_{\text{discard}}(t_i) = w_1 \cdot s_{\text{isolation}}(t_i) + w_2 \cdot s_{\text{edge}}(t_i) + w_3 \cdot s_{\text{pair}}(t_i) + w_4 \cdot s_{\text{jiang}}(t_i)
\end{equation}
where $s_{\text{isolation}}$ penalizes isolated tiles (no $\pm2$ neighbors), $s_{\text{edge}}$ penalizes terminal tiles (1/9), $s_{\text{pair}}$ rewards paired tiles, and $s_{\text{jiang}}$ strongly rewards 2/5/8 pairs. Joker tiles receive $S_{\text{discard}} = -\infty$ (never discarded). If any tile removal results in a ting state, that tile is selected immediately.

\textbf{Interception Decisions.} The \texttt{shouldPeng()} and \texttt{shouldChi()} functions evaluate whether claiming an opponent's discard improves the hand's expected value. A peng is accepted if the resulting hand structure maintains or improves sentence completion probability. A chi is accepted only if it creates a guaranteed sentence without breaking existing pairs.

\subsection{Hand Tile Drag State Management}

The drag-to-reorder system maintains a parallel \texttt{handTileIds} reactive array synchronized with \texttt{handTiles} via Vue~3 watchers. Each tile receives a monotonically incrementing stable identifier, enabling Vue's virtual DOM to track individual tile identity across reorderings. Game operations (draw, discard, peng, chi, gang) synchronously update both arrays: the splice index is computed from the tile array, and the same index is applied to the ID array. In multiplayer mode, a count-based difference algorithm merges server-side tile updates without disrupting client-side ordering.

\subsection{Fault Tolerance Mechanisms}

\textbf{WebSocket Reconnection.} The client implements exponential backoff reconnection with parameters $\tau_0 = 1000$\,ms, $b = 2$, $N_{\max} = 8$ attempts, and ceiling $\tau_{\max} = 120000$\,ms:
\begin{equation}
  \tau_k = \min(\tau_0 \cdot b^k, \tau_{\max}), \quad k = 0,\ldots,N_{\max}-1
\end{equation}
Room state is preserved via URL query parameters \texttt{?auto\_join=ROOMID\&name=PLAYER}, enabling automatic rejoin upon successful reconnection.

\textbf{Server-Side Deadlock Prevention.} The \texttt{checkIntercepts} function sets a 10-second timer upon dispatching action prompts. If the timeout fires before all responders have replied, \texttt{pendingActions} is forcibly cleared and \texttt{nextTurn()} is invoked, ensuring the game never stalls indefinitely due to client disconnection or network issues.

\textbf{Heartbeat Protocol.} Client-side ping messages are sent every 10 seconds. The server immediately responds with a pong. Round-trip time is computed as $\Delta = t_{\text{pong\_recv}} - t_{\text{ping\_sent}}$ and mapped to a 4-tier signal strength indicator displayed in the game UI.

\begin{figure}[H]
\caption{Client-Server Hand Merge Algorithm}
\label{alg:merge}
\footnotesize
\begin{verbatim}
Input:  Client tiles C[], Server tiles S[]
Output: Merged tiles M[] (preserves C order)

1.  fc(v) = count of tile value v in C    // multiset frequency
    fs(v) = count of tile value v in S
2.  R = [], A = []                         // removed, added
    FOR each unique v in C union S:
      IF fc(v) > fs(v): add v to R (fc(v)-fs(v)) times
      IF fs(v) > fc(v): add v to A (fs(v)-fc(v)) times
3.  M = copy of C                          // start from client order
    FOR each v in R: remove first v from M
    FOR each v in A: append v to M
4.  return M
\end{verbatim}

\subsection{iOS-Specific Adaptations}

iOS Safari imposes several unique constraints: (1) \texttt{getUserMedia} requires HTTPS (satisfied by GitHub Pages); (2) AudioContext starts in \texttt{suspended} state and must be resumed within a user-gesture handler; (3) \texttt{click} events only fire on elements with implicit interactive roles (\texttt{<button>}, \texttt{<a href>}); (4) \texttt{position:fixed} behavior is affected by the software keyboard appearance.

All interactive elements in TJMJs UI are implemented as \texttt{<button>} elements with CSS-reset styling. A singleton AudioContext is created on first user interaction and explicitly resumed. Touch event handlers manually manage \texttt{preventDefault()} to suppress iOSs 350ms double-tap-to-zoom delay while preserving tap-to-select functionality.

\subsection{Android Adaptations}

Android Chrome's dynamic address bar causes \texttt{100vh} to include the bar height, resulting in layout shifts when the bar appears/disappears. TJMJ uses the CSS \texttt{100dvh} (dynamic viewport height) unit with a \texttt{100vh} fallback for older browsers. Additionally, Android's fragmented hardware ecosystem leads to variable Agora performance; the Cloud relay infrastructure mitigates this by providing a reliable fallback for connections that fail direct P2P establishment.

\section{Discussion \& Future Work}
% ===================================================================

\subsection{Current Limitations}

Despite extensive engineering, several limitations remain: (1) The Agora Cloud voice topology scales poorly beyond 4 participants, with connection count growing as $\mathcal{O}(N^2)$---introducing an SFU (Selective Forwarding Unit) would reduce this to $\mathcal{O}(N)$; (2) The monolithic \texttt{App.vue} architecture, while efficient for a solo developer, presents maintenance challenges as the codebase grows; (3) The heuristic NPC strategy, while functional, lacks the strategic depth achievable through deep reinforcement learning; (4) Oriental character (★, 🀄, 🌹) rendering in the LaTeX document requires font configuration improvements.

\subsection{Future Research Directions}

\textbf{Neural Mahjong Agent.} Training a deep reinforcement learning agent using Proximal Policy Optimization (PPO) with self-play and Monte Carlo Tree Search (MCTS) to achieve superhuman performance on the Taojiang rule set.

\textbf{Distributed Game Server.} Migrating from a single Node.js process to a horizontally-scalable architecture using Redis pub/sub for cross-instance state synchronization, enabling support for thousands of concurrent rooms.

\textbf{Progressive Web Application.} Implementing Service Worker caching and Web App Manifest for offline-capable, installable mobile experience.

\textbf{Advanced Spectator Features.} Adding replay functionality with time-rewind, multi-perspective auto-switching, and AI-powered commentary generation.

\section{Conclusion}
% ===================================================================

This paper has presented TJMJ, a comprehensive full-stack implementation of the Taojiang classic Wang Mahjong game. The system faithfully implements the complete regional rule set---including dice-driven joker determination, additive pattern scoring, and post-victory multiplier mechanics---within a modern web architecture comprising Vue~3, Node.js, WebSocket, and Agora technologies. Key engineering contributions include a production-grade multiplayer infrastructure with fault tolerance and deadlock prevention, a sub-millisecond recursive hu-detection engine, cross-platform compatibility solutions for iOS and Android, and a real-time voice subsystem leveraging Agora SD-RTN for cellular network traversal. The project serves as both a functional entertainment platform for the Taojiang community and an extensible foundation for mahjong AI research.
