\subsection{背景与动机}

中国麻将是世界上流传最广的桌上博弈游戏之一，其规则在数百年的地域传播中演化出数十种地方变体。湖南桃江地区的"经典王麻将"是其中独具特色的一支：使用精简的 108 张牌组（万/筒/条，不含风牌与花牌）、通过"打筛扳王"动态确定万能牌（癞子）、以"扎鸟"机制在胡牌后翻倍计分、允许多种"门子"（额外胡法）累加番型。这些规则共同塑造了桃江麻将快速、激烈、不可预测的对局节奏，深受当地群众喜爱。

然而，桃江麻将的数字生态长期存在空白。目前市面上仅有闲逸等少数商业 App 提供益阳麻将玩法，但存在以下显著缺陷：\textbf{(1)}~需要强制注册登录，操作链路冗长；\textbf{(2)}~界面设计繁琐，广告干扰严重；\textbf{(3)}~规则实现与桃江本地玩法不完全一致（如缺少扎鸟、扳王等核心机制）；\textbf{(4)}~缺乏拖拽手牌、实时语音、文字弹幕等现代化交互功能；\textbf{(5)}~不开源，无法供二次开发或学术研究使用。此外，微信小程序路径因棋牌类游戏需备案且暂不对个人开放而受阻，Cocos 等游戏引擎入门门槛高、开发效率低。

TJMJ 应运而生——一款完全开源、无需安装、点击链接即用的网页麻将平台。项目在 Vue 3 Composition API + Vite 8 前端框架与 Node.js WebSocket 服务端架构之上，完整实现了桃江麻将的全部规则，并提供单人模式（vs AI）、四人联机实时对战、观战三种玩法模式。本文系该平台的完整技术文档，涵盖系统架构、游戏规则、算法实现、多人联机基础设施、跨平台兼容性方案及性能评测。

\textbf{主要贡献}：
\begin{enumerate}[nosep]
  \item 桃江麻将规则的完整数学形式化——包括牌组编码、骰子力学、癞子计算、操作优先级、计分模型与扎鸟公式；
  \item 生产级全栈架构——服务端权威判定的多人游戏系统，通过 JSON over WebSocket 实现亚 100ms 状态同步延迟；
  \item 递归回溯胡牌检测引擎——$\mathcal{O}(3^k)$ 最坏复杂度，实测 $<$1\,ms，附带听牌检测与 NPC 启发式决策；
  \item 容错多人基础设施——指数退避断线重连、服务端 10 秒操作超时防死锁、URL 参数房间状态持久化；
  \item 全面的跨平台适配方案——解决 iOS Safari 点击事件、AudioContext 挂起、Android 动态地址栏、CSS Transform 破坏 fixed 定位等实际问题。
\end{enumerate}

\subsection{Web 实时游戏技术栈}

TJMJ 的选择基于以下技术成熟度考量：WebSocket（RFC~6455）提供全双工低开销通信，比 HTTP 长轮询减少约 80\% 的延迟开销；WebRTC 标准提供浏览器原生 P2P 音视频能力，配合 STUN/TURN 基础设施可实现跨 NAT 穿透；Vue 3 Composition API 的响应式状态管理天然适合复杂 UI 的增量更新；Vite 8 的原生 ESM 开发服务器和 Rolldown 打包器提供极速冷启动和构建体验。

\subsection{牌组定义与编码}

定义牌组 $\mathcal{T} = \{11,12,\ldots,19,21,\ldots,29,31,\ldots,39\}$。编码规则为 $t = 10s + v$，其中 $s \in \{1,2,3\}$ 表示花色类别（1=万、2=条、3=筒），$v \in \{1,\ldots,9\}$ 表示面值数字。每张牌有 4 个相同的副本，总牌数为 $|\mathcal{T}|=108$。

定义"将"牌集合为 $\mathcal{J} = \{t \in \mathcal{T} \mid t \bmod 10 \in \{2,5,8\}\}$。由于 $s$ 有 3 种、$v$ 有 3 种、每种 4 张，故 $|\mathcal{J}| = 3 \times 3 \times 4 = 36$。将对是胡牌的必要元素。

\subsection{骰子力学与发牌}

开局投掷两枚均匀六面骰子，记点数分别为 $d_1, d_2$（$d_1 \leq d_2$）。总和 $S = d_1 + d_2$ 决定起手墙。从庄家 $P_0$ 开始逆时针点数，$S$ 对应的玩家面前的牌墙即为起手墙（$P_0$ 为 1,5,9；$P_1$ 为 2,6,10；$P_2$ 为 3,7,11；$P_3$ 为 4,8,12）。

开牌位置为起手墙右端开始往左数，跳过 $d_1$ 叠后开始抓牌。发牌过程为 3 轮每人 4 张（共 12 张），随后庄家先补 1 张至 13 张、其他 3 家各补 1 张至 13 张，最后庄家再补 1 张至 14 张。合计 53 张牌被分发。

\subsection{王（癞子）的判定}

从起手墙对应玩家的下家墙左手边开始往右数，到第 $d_2$ 叠处翻开最上面一张牌即为"地"（$T_d$）。"王"（癞子，$T_w$）则按以下公式计算：
\begin{equation}
  T_w = \begin{cases}
    10\cdot\lfloor T_d/10\rfloor + 1, & \text{若 } T_d \bmod 10 = 9 \\
    T_d + 1, & \text{否则}
  \end{cases}
\end{equation}
该循环映射保证了癞子始终与"地"同一花色，数九之后回绕到一。

\subsection{操作优先级模型}

当玩家 $P_s$ 打出牌 $t$ 后，每位其他玩家 $P_i$（$i \neq s$）可能执行拦截操作。合法操作集合为 $\mathcal{A}_i(t) = \{a \in \{\text{HU}, \text{GANG}, \text{PENG}, \text{CHI}\} \mid \text{legal}(i,t,a)\}$。全局优先级排序为 $\text{HU} \succ \text{GANG} \succ \text{PENG} \succ \text{CHI}$。若多名玩家可执行同一操作，以距离打出者逆时针最近者优先。联机模式中，若有多种操作均合法（如既可碰也可吃），全部按钮同时亮起供玩家自主选择。

服务端分发 \texttt{action\_prompt} 后启动 10 秒超时计时器。若超时前相关玩家未全部响应，系统将强制清除挂起状态并推进回合，从而防止因网络中断导致的游戏永久性死锁。

\subsection{胡法分类体系}

一手牌 $H$（14 张）构成胡牌，当且仅当它能分解为\textbf{一对将} $J = \{t_j, t_j\}$（$t_j \in \mathcal{J}$）与\textbf{四句话} $S_1,\ldots,S_4$。每句话可为刻子（$\{t,t,t\}$）或顺子（$\{t,t+1,t+2\}$，同花色）。王（$T_w$）可替代任意牌参与分解。

令 $\text{wang}(H)$ 表示 $H$ 中王的数量。表~\ref{tab:hutype} 列出了所有胡法类型及其评分。

\subsection{计分模型}

自摸得分公式为：
\begin{equation}
  \Sigma = B \cdot \left(\sum_{k} m_k\right) \cdot Z \cdot G
\end{equation}
式中 $B=3$ 为底分，$\sum m_k$ 为所有适用门子的倍数之和（累加制），$Z\in\{1,2\}$ 为扎鸟因子，$G\in\{1,2\}$ 为杠倍因子。

\textbf{扎鸟机制详解：} 胡牌后立即翻开牌墙中本该下一家摸的那张牌（$T_z$），以其个位数值 $v = T_z \bmod 10$ 决定哪些玩家输分翻倍：
\begin{equation}
  Z_i = \begin{cases}
    2, & v \in \{1,5,9\} \quad\text{（全中：三家全部翻倍）}\\
    2, & v \in \{2,6\} \land i = (W+1)\bmod 4 \quad\text{（下家独中）}\\
    2, & v \in \{3,7\} \land i = (W+2)\bmod 4 \quad\text{（对家独中）}\\
    2, & v \in \{4,8\} \land i = (W+3)\bmod 4 \quad\text{（上家独中）}\\
    1, & \text{（未中）}
  \end{cases}
\end{equation}

\textbf{组合胡法实例：}
\begin{itemize}[nosep]
  \item 天胡 + 七小对 + 全中：$(3+2)\times 2 = 10$ 番
  \item 天胡 + 杠上花 + 全中：$3\times 2\times 2 = 12$ 番
  \item 硬庄 + 将将胡 + 碰碰胡 + 天胡 + 杠上花 + 全中：$(2+2+2+2)\times 2\times 2 = 32$ 番
\end{itemize}

核心规则：有王（癞子）时只能自摸，不能抓炮；无王时可自摸也可抓炮。此规则使得"硬庄"更具策略价值。

\subsection{胡牌检测算法}

算法~\ref{alg:hu} 给出了胡牌检测的完整流程。

\begin{figure}[H]
\caption{胡牌检测算法}
\label{alg:hu}
\footnotesize
\begin{verbatim}
输入: 手牌 H (14张), 王牌 Tw, 地牌 Td, 首轮标记 f
输出: {canHu, type, score, hasWang, canCatchCannon}

1.  预处理: N = {t in H | t != Tw}     // 分离普通牌
    nw = |{t in H | t == Tw}|          // 王牌数量
    nd = |{t in N | t == Td}|          // 地牌数量
    Sort(N)                            // 升序排列
2.  特殊拦截: IF f AND |H|=14 AND nw=0:
        IF !hasJiang(H) AND !hasSentence(H):
            RETURN {T, "黑天胡", 2, F, F}
3.  顶级判定: IF nw=4: RETURN {T, "天天胡", 5, T, F}
             IF nw=3: RETURN {T, "天胡", 3, T, F}
             IF nd=3: RETURN {T, "地胡", 2, T, F}
4.  常规判定: normalHu = CheckNormalHu(N, nw)
    is7Pairs = Check7Pairs(N, nw)
5.  无胡: IF NOT normalHu.canHu AND NOT is7Pairs:
            RETURN {F, "", 0, nw>0, !(nw>0)}
6.  门子判定 (按优先级):
    isJiangJiang = all(t in H: t==Tw OR is258(t))
    isQingYiSe = |{floor(t/10), t in N}| == 1
    IF isJiangJiang: RETURN {T, "将将胡", 6, ...}
    IF isQingYiSe:   RETURN {T, "清一色", 6, ...}
    IF is7Pairs:     RETURN {T, "七小对", 6, ...}
    IF isPengPeng:   RETURN {T, "碰碰胡", 6, ...}
    IF nw==0:        RETURN {T, "硬庄", 6, ...}
                     RETURN {T, "平胡", 3, ...}
\end{verbatim}

递归子程序 \texttt{CheckNormalHu(N, w)} 操作如下：取最小牌 $t_{\min}$，依次尝试消除刻子 $\{t_{\min},t_{\min},t_{\min}\}$、顺子 $\{t_{\min},t_{\min}+1,t_{\min}+2\}$、将对子 $\{t_{\min},t_{\min}\}$。普通牌不够时消耗王牌补齐。递归深度 $k\leq 5$，时间复杂度为 $\mathcal{O}(3^k)$，在实际硬件上平均耗时低于 1 毫秒。

\subsection{听牌检测}

一手 $3n+1$ 张牌处于听牌状态，若存在某张 $\mathcal{T}$ 中的牌 $t$ 使得加入后满足胡牌条件。\texttt{RuleChecker.isTing()} 的搜索空间通过限制候补牌仅包含手牌已有花色 $\pm2$ 面值范围来缩减，通常从 34 降至 12--18 个候选。每个候选触发一次完整胡牌检测。

\subsection{NPC 出牌决策}

\texttt{NpcStrategy.js}（227 行）实现启发式 AI：

\begin{enumerate}[nosep]
  \item \textbf{听牌最大化}：遍历每张手牌，模拟丢弃后是否听牌。若能直接听牌，该牌优先级最高。
  \item \textbf{手牌价值评估}：为每张牌计算加权分数——孤张（无相邻牌、非将、非王）得分最低（优先丢弃），依次为边张（1/9）、散搭、对子、将对，王牌得分为负无穷（永不丢弃）。
  \item \textbf{防守策略}：参考弃牌池中已出现的牌，降低对手可能胡的牌的打出的概率。
  \item \textbf{吃碰判断}：\texttt{shouldPeng()} 和 \texttt{shouldChi()} 根据手牌结构改善程度决定是否拦截。
\end{enumerate}

\subsection{手牌拖拽与状态同步}

\textbf{单机模式}：手牌顺序由玩家拖拽直接修改，拖拽操作同步更新 \texttt{handTileIds} 数组以保证 Vue 虚拟 DOM 的稳定键值。新摸的牌追加至数组末尾，不重新排序。所有游戏操作（出牌、吃、碰、杠）均使用 \texttt{splice} 在原数组中精确增删，保持剩余牌的相对顺序。

\textbf{联机模式}：服务端不对手牌排序，客户端的拖拽顺序通过计数差集算法（图~\ref{alg:merge}）与服务端状态合并：比较客户端与服务端的牌频数分布，仅移除多余牌（按客户端位置），新增牌追加至末尾，其余牌保持原顺序。

\begin{figure}[H]
\caption{客户端-服务端手牌合并算法}
\label{alg:merge}
\footnotesize
\begin{verbatim}
输入: 客户端手牌 C[], 服务端手牌 S[]
输出: 合并后的手牌 M[] (保留客户端顺序)

1.  计算频数: fc(v)=C中牌值v的次数, fs(v)=S中牌值v的次数
2.  R={}, A={}   (待移除, 待添加)
    FOR 每个C∪S中的牌值v:
        IF fc(v)>fs(v): v加入R (重复fc(v)-fs(v)次)
        IF fs(v)>fc(v): v加入A (重复fs(v)-fc(v)次)
3.  M = C的副本   (从客户端顺序开始)
    FOR 每个R中的v: 从M中移除第一个匹配项
    FOR 每个A中的v: 追加v到M末尾
4.  RETURN M
\end{verbatim}

\subsection{容错机制}

\textbf{断线重连}：客户端 WebSocket 断开后启动指数退避重连，参数为 $\tau_0 = 1$\,s、$b = 2$、$N_{\max} = 8$、$\tau_{\max} = 120$\,s。房间号通过 URL 查询参数 \texttt{?auto\_join=ROOMID\&name=PLAYER} 持久化，重连成功后自动重新加入原房间。

\textbf{死锁预防}：服务端在分发操作提示后启动 10 秒超时。超时触发时直接清空 \texttt{pendingActions} 并调用 \texttt{nextTurn()}，确保游戏不会因任意玩家的网络故障而永久卡死。

\textbf{心跳与信号}：客户端每 10 秒发送 ping，服务端回复 pong。往返时间 $\Delta = t_{\text{recv}} - t_{\text{sent}}$ 映射为 4 级信号格：$\Delta < 100$\,ms 满格，$\Delta < 200$\,ms 三格，$\Delta < 400$\,ms 两格，$\Delta \geq 400$\,ms 一格，离线为零格。

\textbf{消息节流}：游戏操作（discard, draw, action）间隔至少 400\,ms，WebRTC 信令和聊天消息不节流。连接建立前的消息存入缓冲队列，onopen 后批量冲刷。

\subsection{iOS Safari 适配详解}

iOS Safari 存在若干独特限制：(1) \texttt{getUserMedia} 仅在安全上下文（HTTPS）中可用，GitHub Pages 已满足；(2) AudioContext 在创建后处于 \texttt{suspended} 状态，必须在用户手势回调中显式调用 \texttt{resume()}；(3) 非交互元素（\texttt{<div>}、\texttt{<span>}、\texttt{<img>}）上的 \texttt{@click} 事件不会触发，只有 \texttt{<button>} 和 \texttt{<a href>} 才具有隐式交互角色；(4) \texttt{position:fixed} 在软件键盘弹出时可能表现异常。

TJMJ 将所有交互元素实现为标准 \texttt{<button>}，通过 CSS 重置去除浏览器默认外观（\texttt{-webkit-appearance:none}、\texttt{padding:0}、\texttt{outline:none}）。AudioContext 采用单例模式，首次用户交互时创建并恢复。

\subsection{Android Chrome 适配}

Android Chrome 的动态地址栏会导致 \texttt{100vh} 包含地址栏高度。当地址栏出现或隐藏时，页面高度随之变化。TJMJ 使用 \texttt{100dvh}（动态视口高度）替代，并为不支持该单位的旧浏览器保留 \texttt{100vh} 降级。Android 生态系统的硬件碎片化导致 WebRTC 性能参差不齐，Twilio TURN 中继为 P2P 连接失败的场景提供可靠后备。

\section{讨论与未来工作}
% ===================================================================

\subsection{当前局限性}

\begin{enumerate}[nosep]
  \item \textbf{P2P 网状拓扑扩展性不足}：$N=4$ 人时需管理 $\binom{4}{2}=6$ 条 PeerConnection。引入 SFU 可将复杂度降至 $\mathcal{O}(N)$，并支持更大规模的语音群聊。
  \item \textbf{单文件架构维护成本高}：\texttt{App.vue} 2602 行单体组件在快速迭代阶段效率极高，但随着功能增长，拆分为多个独立组件将更有利于长期维护。
  \item \textbf{启发式 AI 缺乏战略深度}：当前 NPC 策略基于固定的权重规则，无法学习对手风格或适应复杂局面。基于深度强化学习的智能体有望显著提升游戏性。
  \item \textbf{音效素材缺失}：34 个预录制 mp3 文件（27 个牌名 + 7 个动作词）尚未制作，当前全部降级为浏览器 TTS 合成，语音质量和响应延迟均不理想。
  \item \textbf{LaTeX 字体限制}：手册中部分 Unicode 符号（★、🀄、🌹）和方框绘图字符（├─└）在当前等宽字体配置下无法正确渲染，未来可通过更换字体或改用 TiKZ 绘图解决。
\end{enumerate}

\subsection{未来研究方向}

\begin{itemize}[nosep]
  \item \textbf{深度强化学习麻将智能体}：采用 PPO 算法在自我对弈环境下训练，结合蒙特卡洛树搜索 (MCTS) 提升决策质量，目标达到或超越人类高手水平。
  \item \textbf{分布式游戏服务器}：引入 Redis Pub/Sub 实现跨 Node.js 进程的广播同步，配合负载均衡器实现水平扩展，支持数千个并发房间。
  \item \textbf{渐进式 Web 应用 (PWA)}：通过 Service Worker 实现离线缓存、推送通知和主屏安装，提供接近原生应用的移动体验。
  \item \textbf{回放系统与智能观战}：记录每局完整操作日志，支持时序回放、多视角自动切换，以及基于 GPT 的对局解说生成。
  \item \textbf{跨平台桌面客户端}：使用 Electron 或 Tauri 封装 Web 前端为桌面应用，实现系统级快捷键、通知和更好的性能。
\end{itemize}

\subsection*{A.2 组合胡法计算公式}

组合胡法自摸得分通用公式：
\begin{equation}
S_{\text{自摸}} = (\Sigma\,\text{门子倍数}) \times F_{\text{扎鸟}} \times F_{\text{杠}}
\end{equation}

\textbf{实例}：
\begin{itemize}[nosep]
  \item 天胡七小对全中：$(3+2)\times 2 = 10$ 番
  \item 天胡杠上花全中：$3\times 2\times 2 = 12$ 番
  \item 硬庄将将胡不中：$2\times 2\times 2 = 4$ 番
  \item 软庄碰碰胡的地胡带拖不中：$(2+2)\times 1\times 1 = 4$ 番
  \item 硬庄将将胡的碰碰胡的天胡杠上花全中：$(2+2+2+2)\times 2\times 2 = 32$ 番
\end{itemize}

\subsection*{A.3 补充胡法规则}

\begin{enumerate}
  \item \textbf{诈胡}：牌局结束，由诈胡方当庄家，看牌型应得多少赔多少给三家。
  \item \textbf{起手报听}：开局摸牌前，除庄家外其他三家起手 13 张只差一个"将"或"一句话"即可听牌，有王可以抓炮。
  \item \textbf{天胡七小对}：3 番 + 2 番 = 5 番；天胡杠上花：3 番 $\times$ 2 $\times$ 2 = 12 番。
  \item \textbf{明杠与暗杠}：暗杠手中有四张相同牌不用亮出；明杠别人打出一张你手中有三张。开杠后重新掷骰子从牌墙末尾摸三张。抢杠规则：明杠时该牌若恰好是其他玩家要胡的牌，直接判杠家全赔。
  \item \textbf{112 牌规则}：108 张主牌（万筒条各 36 张）+ 36 张将（数字为 2/5/8 的牌）= 共 144 张麻将牌中实际使用 108 张。
\end{enumerate}
